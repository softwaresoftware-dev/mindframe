MINDFRAME — INSTALL INSTRUCTIONS FOR THE AGENT
================================================

You are the install agent. A real human launched Claude Code and pasted this
into the chat. Your job is SMALL and BOUNDED: install the bundle, then birth
the operator's first mindframe and hand setup over to it. You do NOT run a
long terminal wizard. The setup conversation happens INSIDE a web surface that
a mindframe agent drives — installing mindframe is itself the operator's first
experience of a mindframe.

End state of THIS document: the bundle is installed, a setup mindframe is
running, its surface is open in the operator's browser, and you have stepped
out of the way. Everything else (identity, schema, connections, the knowledge
base, the first signal) happens in that surface, not here.

==============================================================================
PHASE 0 — RULES YOU MUST FOLLOW THROUGHOUT
==============================================================================

CONVERSATION, NOT SCRIPT
  Stop and ask when uncertain. Surface evidence for every claim. Never
  fabricate. If a probe fails, ask the operator — do not retry blindly.

INHERITED IDENTITY, NO TOKEN COLLECTION
  Mindframe runs as the operator. Anything they can do via gh / az / gcloud /
  aws / a loaded MCP, it does by invoking those same tools. Never ask for a PAT
  an existing CLI already holds. If a CLI is unauthed, ask the operator to run
  the provider's own login (`gh auth login`, etc.) — a browser OAuth handoff.

GENERATED SECRETS ARE FILE-HANDOFF, NEVER CHAT
  Any token you create goes to a file under ~/.mindframe/secrets/ (chmod 700
  dir, chmod 600 file). Print the path and a clipboard command. Never print the
  value.

IDEMPOTENCY + RESUMABILITY
  Every phase detects "already done" and skips. Derive done-ness from the WORLD
  first — plugin in cache, vault dir exists, daemon green on /api/health, agent
  already registered. World-state probes are authoritative.

USER-SCOPE BY DEFAULT
  Every MCP or plugin you enable is user-scope. Never project-scope.

NO ANTHROPIC API KEY
  Mindframe runs on the Claude Code subscription. Never ask for one.

==============================================================================
PHASE 0.W — WINDOWS OPERATORS ONLY (skip if Linux or macOS)
==============================================================================

Mindframe runs on Linux + macOS natively; Windows runs inside WSL2 (taskpilot's
spawner needs tmux). Detect WSL: $(uname -r) contains "microsoft" OR
$WSL_DISTRO_NAME is set -> proceed to PHASE 1. Otherwise the operator is on
native Windows: walk them through `wsl --install Ubuntu`, then inside WSL
install curl/tmux/git/python3/python3-pip + node + @anthropic-ai/claude-code,
relaunch claude inside WSL, and re-paste this URL there. Keep one WSL shell
open (or enable systemd + a long shutdown timeout) so the daemons survive, and
prefer notify-slack / notify-email over notify-linux (no notify-send in WSL).

==============================================================================
PHASE 1 — BOOTSTRAP THE MARKETPLACE + RESOLVER
==============================================================================

First ensure `uv` is installed — the softwaresoftware resolver runs its MCP
server via `uv` (`uv run python server.py`). Without it the resolver silently
fails to connect and PHASE 2 dead-ends. This is a no-op if uv is already present:

  command -v uv >/dev/null 2>&1 || curl -LsSf https://astral.sh/uv/install.sh | sh

(uv installs to ~/.local/bin — make sure that's on PATH for this session, e.g.
`export PATH="$HOME/.local/bin:$PATH"`.)

Then run via Bash (both idempotent):

  claude plugin marketplace add softwaresoftware-dev/softwaresoftware-plugins
  claude plugin install softwaresoftware@softwaresoftware-plugins

Then ask the operator to type `/reload-plugins` so the running session picks up
the softwaresoftware skill. /reload-plugins has no Bash equivalent — it is the
one unavoidable manual step in the bootstrap. Continue in the same session
after they reload; no re-paste needed.

==============================================================================
PHASE 2 — INSTALL MINDFRAME + DEPENDENCIES
==============================================================================

Invoke the softwaresoftware install skill (this is a skill in the running
session, NOT a shell command — never run it via Bash). If you can invoke
skills directly, invoke `softwaresoftware:install` with argument `mindframe`;
otherwise ask the operator to type:

  /softwaresoftware:install mindframe

This resolves the capability graph (agent-spawning, session-mesh, event-routing,
browser-automation, daemon) and picks providers that match the environment.
After install, run /reload-plugins.

Verify: mindframe + every dependency present in ~/.claude/plugins/cache/.
The agent-spawning provider's daemon (the thing that supervises agents and
delivers messages to them) must be running — confirm its health endpoint
answers before PHASE 3.

==============================================================================
PHASE 3 — STAND UP THE STACK + BIRTH THE FIRST WORKSPACE
==============================================================================

Mindframe is ONE shared stack that serves many workspaces. You stand that stack
up once, create the operator's first workspace ("personal") as an isolated
partition, spawn the setup mindframe INTO it, and open the surface. There are NO
per-workspace daemons and no per-workspace ports.

3.0 — THE MODEL (what you are standing up). One daemon each, all reading
  ~/.mindframe:
    session-bridge       :8910   mesh (session-mesh provider)
    taskpilot            :8912   agent runtime — per-task HOME isolates workspaces
    dispatcher           :8911   ingress + poller (event-routing provider)
    mindframe-dashboard  :5174   the Surface — multi-tenant; mindframe owns it
  A workspace is a PARTITION dir, never a deployment:
    ~/.mindframe/workspaces/<id>/
      .mindframe/{vault,frames,connections,secrets,
                  dispatcher/{channels.yaml,recipes,event-sources}}
      .claude/{skills, settings.json, .credentials.json, plugins}
      .claude.json
  Registry: ~/.mindframe/workspaces.yaml  ->  {workspaces: {<id>: {label}}}
  The dashboard serves the workspace PORTAL at / and each workspace's home at
  /w/<id>/ ; it derives frames/vault per request from
  ~/.mindframe/workspaces/<id>/.mindframe/, so do NOT set MINDFRAME_FRAMES_ROOT
  or MINDFRAME_VAULT_DIR.

3.1 — CREATE THE FIRST WORKSPACE ("personal"). A partition + a seeded login so
  its agents run on the operator's Claude subscription with NO per-workspace
  OAuth. (A fresh HOME otherwise runs Claude Code's first-run onboarding, whose
  first step is sign-in — `hasCompletedOnboarding` is the key that skips it.)

    WS="$HOME/.mindframe/workspaces/personal"
    mkdir -p "$WS/.mindframe"/{vault,frames,connections,secrets} \
             "$WS/.mindframe/dispatcher"/{recipes,event-sources} \
             "$WS/.claude/skills"
    chmod 700 "$WS/.mindframe/secrets"
    printf 'routes: []\npaused_routes: []\n' > "$WS/.mindframe/dispatcher/channels.yaml"
    printf 'workspaces:\n  personal:\n    label: Personal\n' > "$HOME/.mindframe/workspaces.yaml"

    # share subscription auth + plugin code; keep MCPs / skills / vault local
    ln -sfn "$HOME/.claude/.credentials.json" "$WS/.claude/.credentials.json"
    ln -sfn "$HOME/.claude/plugins"           "$WS/.claude/plugins"

    # share OS-level CLI identity (one machine identity) so the agent's
    # gh / gcloud / aws / git keep working under HOME=$WS. Per-workspace
    # difference lives in MCPs, connector skills, and the vault — not the
    # base CLIs.
    for id in .gitconfig .npmrc .ssh .config .aws .azure .gnupg; do
      [ -e "$HOME/$id" ] && ln -sfn "$HOME/$id" "$WS/$id"
    done

    # per-workspace settings.json: carry plugin enablement / hooks / permissions
    # but NOT global mcpServers (they would leak into every workspace)
    python3 - "$HOME/.claude/settings.json" "$WS/.claude/settings.json" <<'PY'
import json,sys
try: d=json.load(open(sys.argv[1]))
except Exception: d={}
d["mcpServers"]={}
json.dump(d,open(sys.argv[2],"w"),indent=2)
PY

    # .claude.json: empty mcpServers (isolated) + the auth/onboarding state so a
    # spawned agent here is already signed in on the subscription
    python3 - "$HOME/.claude.json" "$WS/.claude.json" <<'PY'
import json,sys
try: r=json.load(open(sys.argv[1]))
except Exception: r={}
seed={"mcpServers":{}}
for k in ("enabledPlugins","oauthAccount","userID","hasCompletedOnboarding","lastOnboardingVersion","numStartups"):
    if k in r: seed[k]=r[k]
json.dump(seed,open(sys.argv[2],"w"),indent=2)
PY

3.2 — SETUP FRAME DIR, inside the personal partition (frame id == agent task id
  so message delivery aligns):
    FRAME_DIR="$WS/.mindframe/frames/mindframe-setup"
    mkdir -p "$FRAME_DIR"
    printf '{"id":"mindframe-setup","title":"Setup","task_id":"mindframe-setup","status":"active"}\n' > "$FRAME_DIR/meta.json"

3.3 — LOCATE THE PLUGIN ASSETS:
    MF_ROOT="$(ls -d "$HOME"/.claude/plugins/cache/*/mindframe/*/ 2>/dev/null | sort -V | tail -1)"
    # brief: $MF_ROOT/setup/brief.md   dashboard: $MF_ROOT/dashboard/server/server.py

3.4 — FILL THE SETUP BRIEF. The agent runs with HOME=$WS, so its vault is
  $WS/.mindframe/vault. Substitute the frame dir:
    sed -e "s#__FRAME_DIR__#$FRAME_DIR#g" "$MF_ROOT/setup/brief.md" > "$FRAME_DIR/brief.txt"
  That filled-in text IS the setup mindframe's standing brief.

3.5 — SPAWN THE SETUP MINDFRAME into the personal partition. Use an available
  agent-spawning tool to create a LONG-RUNNING agent (idles between messages,
  stays reachable, never declares itself done — NOT a fire-and-forget batch job):
    name:        mindframe-setup
    working dir: $FRAME_DIR
    HOME:        $WS    <- the workspace partition (per-task home). This isolates
                           its MCPs / skills / vault and runs it on the seeded
                           subscription login. If the spawning tool exposes a
                           per-task home/HOME option, set it to $WS.
    brief:       the contents of $FRAME_DIR/brief.txt
  On startup it composes its first page (BEAT 1: "this is you"). Wait until
  $FRAME_DIR/index.html exists before 3.7.

3.6 — RUN THE STACK, workspace-aware. session-bridge / taskpilot / dispatcher
  were installed by PHASE 2's providers; mindframe composes two pieces of
  workspace wiring on top:

  a) DISPATCHER POLLER + INGRESS must discover each workspace's
     event-sources/channels. Ensure their env has
       DISPATCHER_WORKSPACES_ROOT=$HOME/.mindframe/workspaces
     and NO ANTHROPIC_API_KEY (subscription only). If the provider's setup didn't
     set these, re-register the dispatcher daemons through the daemon tool with
     them.

  b) THE DASHBOARD (mindframe owns it) — run as a managed daemon
     (reboot-persistent). One server serves the portal at /, every workspace at
     /w/<id>/, and each mindframe's surface at /w/<id>/m/<frame>:
       command: $VENV/bin/python "$MF_ROOT/dashboard/server/server.py"
       env: PORT=5174
            MINDFRAME_HOME=$HOME/.mindframe
            MINDFRAME_TASKPILOT_DAEMON=<taskpilot base url, e.g. http://127.0.0.1:8912>
            MINDFRAME_DISPATCHER_URL=<dispatcher base url, e.g. http://127.0.0.1:8911>
       Do NOT set MINDFRAME_FRAMES_ROOT / MINDFRAME_VAULT_DIR (derived per
       workspace). Ensure ANTHROPIC_API_KEY is absent from the daemon env.
     venv (PEP 668 — never `pip install` globally):
       VENV="$HOME/.mindframe/dashboard-venv"
       python3 -m venv "$VENV"
       "$VENV/bin/pip" install --quiet fastapi uvicorn httpx pyyaml

3.7 — OPEN THE SURFACE. Open the operator's browser to the setup mindframe:
       http://127.0.0.1:5174/w/personal/m/mindframe-setup
  (the portal of all workspaces is at http://127.0.0.1:5174/.) They should see a
  single glowing "you" node and an invitation to say what they do.

==============================================================================
PHASE 4 — HAND OFF
==============================================================================

Your terminal job is done. Tell the operator, plainly:

  "Mindframe is installed and your first workspace is open at
   http://127.0.0.1:5174/w/personal/m/mindframe-setup . Setup continues there —
   just talk to it in the message box. Tell it what you do and it'll build your
   knowledge base in front of you. Your other workspaces live at
   http://127.0.0.1:5174/ . I'm done here in the terminal."

Do NOT keep narrating phases in the chat. The setup mindframe owns the rest.
If the operator comes back to the terminal with a problem, help; otherwise the
conversation has moved into the surface.

==============================================================================
WHAT THE SETUP MINDFRAME DOES (context — you don't run this; it does)
==============================================================================

The spawned mindframe-setup agent owns one HTML surface it rewrites in place,
driven by the operator's message box. Its arc:

  1. THIS IS YOU      seed the operator's Person node from their git/gh
                      identity; render one node + empty schema/connections rails.
  2. INTERVIEW        "what do you do, and who for?" -> derive entity TYPES ->
                      schema.yaml; legend chips appear `pending`. No data nodes.
  3. DISCOVER         claude mcp list + gh/gcloud/aws auth probes (against the
                      operator's real home) -> connections rail.
  4. SYNTHESIZE       pull the strongest reachable source -> interpret into
                      schema-valid entities across shapes -> nodes bloom, chips
                      light, scope reported honestly.
  5. FIRST SIGNAL     one real, actionable thing from the connected source.
  (later) configure the human-in-the-loop away-path, author the first scheduled
          agent, author the first event-source agent.

Capabilities are skills / loaded MCPs / the operator's CLIs — discovered via
shell, not a catalog, and not stored in the knowledge base. The knowledge base
holds what the org IS; capabilities are how the agent acts.
